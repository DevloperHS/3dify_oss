**Ticket:** `01-research-reconstruction-approach.md` · **Scope:** compare candidate approaches for the Reconstruction pipeline stage (single image → 3D mesh), for a bootstrapped, multi-user, async, GLB-output production service.

## Meshy

**Cost model.** Web app has a Free plan ($0, 100 credits/month, CC BY 4.0-licensed outputs, no API access) and a Pro plan ($20/mo or $240/yr, 1,000 credits/month, API access included, 10 concurrent tasks) and a Studio plan ($60/mo or $576/yr) ([meshy.ai/pricing](https://www.meshy.ai/pricing)). The API itself is "pay-before-you-go": credits are purchased separately from the subscription. Image-to-3D costs, per the API pricing docs: "Meshy-6 models and low poly models: 20 credits (without texture), 30 credits (with texture); Other models: 5 credits (without texture), 15 credits (with texture)" ([docs.meshy.ai/en/api/pricing](https://docs.meshy.ai/en/api/pricing)). The dollar-per-credit conversion for API credits specifically was not stated on the fetched page — only the credits-per-call figures were confirmed as primary-sourced.

**Output quality.** Image-to-3D API supports GLB as its always-on primary output format, plus optional FBX/OBJ/USDZ/STL/3MF; base color textures up to 4K, PBR maps (metallic/roughness/normal/emission) at 2K; polycount is controllable from 100 to 300,000 with adaptive decimation ([docs.meshy.ai/en/api/image-to-3d](https://docs.meshy.ai/en/api/image-to-3d)). No published quantitative mesh-fidelity benchmark (Chamfer distance, F-score, etc.) was found — Meshy is a commercial product, not an academic release, and doesn't publish this kind of number.

**Latency.** "Most image-to-3D conversions complete in about one minute with the latest Meshy 6 model" ([meshy.ai/features/image-to-3d](https://www.meshy.ai/features/image-to-3d)). Enabling multi-view thumbnail rendering "adds approximately 3 seconds to task latency" ([docs.meshy.ai/en/api/image-to-3d](https://docs.meshy.ai/en/api/image-to-3d)).

**Data-handling.** From the Terms of Use ([meshy.ai/terms-of-use](https://www.meshy.ai/terms-of-use)):
- "Meshy may use Customer Inputs and Customer Outputs from non-Enterprise Customers... to train, validate, test, or improve Services unless otherwise agreed to in the Order" — this applies to free-tier use.
- Paid-plan customers grant Meshy a license to process content "as may be necessary... to provide the Service," and paid customers "have the option to keep their User Content private," with content "not... used for any purpose other than as outlined here."
- **API-specific, non-Enterprise:** "Customer Output generated by Customers using the APIs, other than Enterprise Customers, will be deleted three (3) days after it is generated."
- Enterprise customers get configurable retention (1–30 days, or indefinite by default) and the agreement text found made no mention of Enterprise data being used for training.
- Data is stored on AWS in the US; Meshy states ISO/IEC 27001:2022 and SOC 2 certification ([meshy.ai/privacy-policy](https://www.meshy.ai/privacy-policy)).

This means: on the Pro/API tier (the one this project would actually use), user-uploaded images are not stated to be used for training, and generated outputs are auto-deleted after 3 days — good for a "no hard privacy constraint but must be known" project, since it caps exposure automatically.

## Tripo3D / Tripo AI

**Cost model.** API credits are $0.01 each. Image-to-model: H2/H3 models cost 20 credits without texture / 30 with texture ($0.20 / $0.30); the P1 model costs 40/50 credits ($0.40/$0.50). New accounts get "300 free credits in 2 weeks." Premium options (detailed texture, smart low-poly, quad output) add 5–20 credits ([docs.tripo3d.ai/get-started/pricing.html](https://docs.tripo3d.ai/get-started/pricing.html)). Separately, the Tripo Studio (consumer web app) has its own Free (300 credits/mo) and Professional ($19.90/mo, 3,000 credits/mo) plans, which are billed independently from API credits per Tripo's own docs description of the split ([tripo3d.ai/pricing](https://www.tripo3d.ai/pricing)).

**Output quality.** The developer/API marketing page (developers.tripo3d.ai) and the platform docs (platform.tripo3d.ai) are JavaScript-rendered single-page apps; my fetcher could only retrieve the page `<title>`, not body content, on multiple attempts. I could not independently confirm GLB output or a mesh-fidelity benchmark from Tripo's own docs in this research pass — third-party API resellers (fal.ai, WaveSpeedAI) list GLB as a supported output for Tripo models, but per the task's instruction I'm not treating a reseller's listing as a Tripo-primary source. **This should be re-verified with a JS-capable browser before committing to Tripo.**

**Latency.** Tripo's own docs page state generation "typically completes within a few seconds" once out of the task queue ([platform.tripo3d.ai/docs/generation](https://platform.tripo3d.ai/docs/generation), title/snippet only retrievable). A third-party host (WaveSpeedAI) reports ~71 seconds average end-to-end including their own queueing — not a Tripo-primary number, included only as a rough sanity check, flagged as such.

**Data-handling.** From the Terms of User Agreement ([tripo3d.ai/terms](https://www.tripo3d.ai/terms)):
- Free users (§5.2.1): Tripo "retains all rights, including without limitation the rights to use, copy, reproduce, modify, adapt, publish, translate, create derivative works from, distribute" their inputs/outputs — effectively unrestricted use, consistent with training use.
- Paid users (§5.2.2): "Company will not use Inputs and Outputs as training data to train, validate, test, or improve any AI Technology."
- §4: "Tripo has no obligation to store any of Inputs and Outputs that you make available on Tripo Properties" — i.e., no guaranteed retention, but also no guaranteed deletion window stated.
- §10.5: content previously posted to "public" areas "may be retained in perpetuity" — not directly relevant to a private API integration, but worth noting if any public gallery feature is ever used.

This is a materially better data-handling story than the free tier of most competitors, provided the project uses a paid/API key (not free-tier), and it's simple: don't use their free tier for real user images.

## Stability AI — Stable Fast 3D

**Cost model.** Stability's Platform API uses a credit system where "1 credit = $0.01" and new accounts get 25 free credits ([search-derived summary of platform.stability.ai/pricing](https://platform.stability.ai/pricing) — the page itself is JS-rendered and my fetcher could only retrieve the header, not the pricing table body, so treat the exact per-call credit cost as **not independently confirmed** by me: one aggregator source states 2 credits/call ($0.02), another states 10 credits/call ($0.10). **This numeric conflict needs to be resolved directly against the live platform.stability.ai pricing table (which requires a JS-capable fetch) before this candidate is used for cost modeling.**

**Output quality.** Stable Fast 3D (SF3D) is a 1B-parameter transformer, trained on Objaverse renders, that produces a "UV-unwrapped textured 3D mesh asset" with low polygon count and per-object PBR material parameters (roughness, metallic), with delighting preprocessing applied; input is 512×512 images ([HF model card](https://huggingface.co/stabilityai/stable-fast-3d)). No Chamfer/F-score numbers are given on the model card itself — those would be in the associated paper (not separately verified here). License: Stability AI Community License, free for orgs/individuals earning ≤$1M/year revenue, Enterprise License required above that ([HF model card](https://huggingface.co/stabilityai/stable-fast-3d)).

**Latency.** "create a 3D model from a single image in under one second," and elsewhere "in 0.5 seconds" ([HF model card](https://huggingface.co/stabilityai/stable-fast-3d)). Note this model is also available as open weights (see below) — it's simultaneously a candidate for the API path and the self-hosted path, which is worth keeping in mind since self-hosting removes the per-call cost entirely.

**Data-handling.** Stability's Platform API Terms of Service, §4(c): Stability may use Inputs/Outputs to "improve and develop our Services (but you can opt-out to prevent us from using your Inputs and Outputs to train our models)" ([stability.ai/terms-of-service](https://stability.ai/terms-of-service)). A dedicated knowledge-base article confirms the opt-out is a toggle in account settings and that by default "your content [is] utilized in training our models" ([kb.stability.ai — Opt-out of Data Training for Platform API](https://kb.stability.ai/knowledge-base/opt-out-of-data-training-for-platform-api)). General privacy policy states personal information is retained "for as long as necessary," capped at one year unless law requires longer ([stability.ai/privacy-policy](https://stability.ai/privacy-policy) — general policy, not API-specific).

Net: default behavior is opt-out (training happens unless you flip a setting) — the opposite default of Tripo's paid tier. Workable, but requires the project to actively flip the opt-out toggle and document that it did so.

## Hyper3D Rodin (Deemos)

Verified as a real, currently operating commercial product as of this research date (July 2026); included per the ticket's "other credible commercial API" allowance.

**Cost model.** Free tier: $0/mo, includes free-before-confirmation generation, 10 private assets, basic image-to-3D. Pay-as-you-go: "$1.50/credit." Creator plan: $30/mo ($24/mo billed yearly at $288/yr), ~60 models/month estimated. Business plan: $120/mo ($96/mo billed yearly at $1,152/yr), ~416 models/month estimated, includes full API access, 4K textures, 120–240 RPM API rate limit. Enterprise: custom ([hyper3d.ai/pricing](https://hyper3d.ai/pricing)). On the Gen-2.5 API specifically, quality tiers cost 0.5–1.0 credits each ([Rodin Gen-2.5 API docs](https://developer.hyper3d.ai/api-specification/rodin-gen2.5.md)) — at $1.50/credit that's $0.75–$1.50 per generation, notably more expensive per-call than Meshy or Tripo.

**Output quality.** GLB is explicitly supported (`geometry_file_format` parameter, default `glb`), alongside USDZ/FBX/OBJ/STL. Five mesh-density presets from Extreme-Low to Extreme-High, with a `quality_override` allowing 1,000–2,000,000 faces (quad mode capped lower, at 200,000). Texture options: PBR, Shaded (baked lighting), both, or none; optional HD-texture add-on for 4K ([developer.hyper3d.ai/api-specification/rodin-gen2.5.md](https://developer.hyper3d.ai/api-specification/rodin-gen2.5.md)). No published Chamfer/F-score benchmark found.

**Latency.** Not found in the primary docs pages fetched — **unverified**. Would need to be measured directly or found in docs not surfaced by this research pass.

**Data-handling.** Deemos' own developer-docs data policy states data is "securely stored for 7 days, will not be used for training purposes, and will not be shared without explicit consent" ([developer.hyper3d.ai/legal/data-policy](https://developer.hyper3d.ai/legal/data-policy) — page content confirmed via search snippet, not a full independent fetch, so treat as high-confidence but not double-verified). This is the strongest stated privacy commitment of the three flagship commercial APIs researched (no training by default, bounded 7-day retention), if confirmed on direct read of the policy page.

## TripoSR (self-hosted)

**Cost model.** MIT-licensed, free to self-host, commercial use permitted ([GitHub](https://github.com/VAST-AI-Research/TripoSR)). README states "~6GB VRAM for a single image input" for default inference. That fits comfortably on a single mid-tier consumer/cloud GPU. On RunPod, an RTX 4090 (24GB) is $0.69/hr and an RTX A6000 (48GB) is $0.49/hr ([runpod.io/pricing](https://www.runpod.io/pricing)) — either is heavily over-provisioned for a 6GB workload, meaning a smaller/cheaper GPU tier (not itemized on RunPod's page as fetched) would likely suffice; this is the cheapest inference footprint of any candidate researched.

**Output quality.** Published in the TripoSR paper (Tochilkin et al., arXiv:2403.02151): on Google Scanned Objects (GSO), Chamfer Distance 0.111 and F-score@0.1 0.651; on OmniObject3D, CD 0.102 and F-score@0.1 0.677 — beating the compared open baselines (OpenLRM, TGS) on both metrics at time of publication ([arxiv.org/abs/2403.02151](https://arxiv.org/abs/2403.02151)). Default output is vertex-colored geometry; textured output requires an explicit `--bake-texture` flag with configurable texture resolution ([GitHub README](https://github.com/VAST-AI-Research/TripoSR)). Note InstantMesh's own paper later benchmarks TripoSR less favorably on the same GSO set using a different eval protocol (see InstantMesh section) — the two papers are not directly comparable since evaluation setups differ; both numbers are reported as each paper states them, not reconciled.

**Latency.** "less than 0.5 seconds on an NVIDIA A100 GPU" — stated on both the GitHub README and in the paper itself ([GitHub](https://github.com/VAST-AI-Research/TripoSR), [arXiv:2403.02151](https://arxiv.org/abs/2403.02151)).

**Data-handling.** N/A — self-hosted, so the project fully controls image retention/deletion, satisfying "no hard privacy constraint, but must be known" trivially (it's whatever the project builds).

## InstantMesh (self-hosted)

**Cost model.** Apache-2.0 licensed, free to self-host ([HF model card](https://huggingface.co/TencentARC/InstantMesh)). VRAM requirement is not stated as a number in the README. A filed GitHub issue documents a user OOM-ing while running the Gradio demo on a 24GB RTX 4090 (attempted to allocate 15GB against ~12.5GB free after ~10GB already in use) ([GitHub issue #126](https://github.com/TencentARC/InstantMesh/issues/126)) — this is real-world evidence, not an official spec, but it indicates the full pipeline (multiview diffusion + LRM reconstruction) needs meaningfully more headroom than TripoSR, likely pushing toward a 40–80GB card for comfortable production use. On RunPod, A100 80GB is $1.39–1.49/hr ([runpod.io/pricing](https://www.runpod.io/pricing)) — roughly 2–3x TripoSR's likely GPU cost per hour, though InstantMesh's higher fidelity may still make it the better cost/quality tradeoff per generation depending on batch throughput.

**Output quality.** Published in the InstantMesh paper (Xu et al., arXiv:2404.07191): on GSO orbiting-view eval, InstantMesh (Mesh variant) scores PSNR 22.794 / SSIM 0.897 / LPIPS 0.120 / CD 0.180 / F-score 0.880; the NeRF variant scores PSNR 23.141 / SSIM 0.898 / LPIPS 0.119 / CD 0.177 / F-score 0.882. In the same table, TripoSR scores PSNR 23.373 / SSIM 0.868 / LPIPS 0.213 / CD 0.217 / F-score 0.843, and CRM scores PSNR 22.195 / SSIM 0.891 / LPIPS 0.150 / CD 0.252 / F-score 0.787 — i.e. InstantMesh beats both on CD and F-score, is close on PSNR ([arxiv.org/abs/2404.07191](https://arxiv.org/abs/2404.07191)). Default output is a textured-via-vertex-color `.obj` mesh; a texture map (UV-unwrapped) is available via `--export_texmap`, at the cost of longer processing ([GitHub README](https://github.com/TencentARC/InstantMesh)). Trained on 8× NVIDIA H800 GPUs (training hardware, not inference requirement) ([paper](https://arxiv.org/abs/2404.07191)).

**Latency.** "within 10 seconds" total, combining the multiview diffusion step and sparse-view reconstruction ([GitHub](https://github.com/TencentARC/InstantMesh), [arXiv:2404.07191](https://arxiv.org/abs/2404.07191)) — an order of magnitude slower than TripoSR's sub-second figure, but still fast in absolute terms for an async worker pipeline.

**Data-handling.** N/A — self-hosted.

## Recommendation

**Top pick: TripoSR, self-hosted, as the reconstruction worker.**

Reasoning tied to this project's actual constraints (see `map.md`):
1. **Cost-sensitive/bootstrapped.** TripoSR is MIT-licensed and free to run; its ~6GB VRAM footprint ([GitHub README](https://github.com/VAST-AI-Research/TripoSR)) is the cheapest inference profile of every candidate researched — well below even the smallest GPUs itemized on RunPod's pricing page. At sub-$1/hr GPU rental and sub-second inference, marginal cost per generation is a small fraction of a cent in compute, versus $0.20–$1.50 per call for every commercial API researched. For a service where usage-based billing doesn't exist yet (per `map.md`, billing is out of scope for this spec) but the founder is paying real cloud bills from day one, this matters a lot.
2. **Fits the worker-based async pipeline.** `map.md` already specifies "a separate worker process for Reconstruction (scales independently, GPU-capable)" — that's exactly the shape self-hosting TripoSR wants: load model once, keep a GPU worker warm, pull jobs off BullMQ, run inference, push a GLB to R2. No external network dependency in the hot path (aside from your own GPU worker), no third-party rate limits or outages to plan around.
3. **GLB fits.** TripoSR's default output isn't GLB natively (vertex-colored mesh, optional baked texture) — this needs a conversion step (e.g. via `trimesh`) in Postprocessing, which the pipeline already has as a distinct stage per `map.md`. Not a blocker, but worth flagging as an explicit task in that stage's ticket.
4. **No hard privacy constraint, but data-handling must be known** — self-hosting makes this a non-issue: there is no third party to have terms with. This directly satisfies the constraint by removing the question.
5. **Quality is real, not just fast.** Published GSO/OmniObject3D numbers ([arXiv:2403.02151](https://arxiv.org/abs/2403.02151)) show it's competitive with (not the best of) contemporaneous open methods — InstantMesh's own paper shows InstantMesh beating it on the same axes.

**Second pick, worth prototyping in parallel: InstantMesh, self-hosted.**

If TripoSR's mesh fidelity proves visibly worse in early user testing (it emphasizes speed over max quality, and its texture story is the weaker of the two — vertex-color by default, baked texture optional, vs. InstantMesh's UV-unwrapped texture map), InstantMesh is the natural upgrade path: same self-hosted cost model (no per-call fee), same "no third party" privacy answer, still fast enough for an async worker (~10s vs TripoSR's <0.5s — both are fine when the user is already waiting on a queued job, not a synchronous request), and it beats TripoSR on CD/F-score in InstantMesh's own published comparison ([arXiv:2404.07191](https://arxiv.org/abs/2404.07191)). The tradeoff is a heavier GPU requirement — real-world evidence points to needing meaningfully more than 24GB VRAM ([GitHub issue #126](https://github.com/TencentARC/InstantMesh/issues/126)), which roughly doubles or triples the GPU rental cost per hour (A100 80GB at $1.39–1.49/hr vs. a 24GB-class card at $0.49–0.69/hr, per [RunPod pricing](https://www.runpod.io/pricing)) — still trivial next to any commercial API's per-call pricing.

**Why not a commercial API as the primary pick.** Every commercial API researched (Meshy, Tripo3D, Stability, Hyper3D) charges per generation ($0.20–$1.50+), which directly fights the "cost-sensitive/bootstrapped" constraint at any meaningful volume, and each has its own data-handling regime that has to be actively managed (Meshy's 3-day API auto-delete and free/paid training split; Tripo's free-vs-paid training split; Stability's opt-out-required default; Hyper3D's stated no-training-by-default). None of that is disqualifying — Tripo3D's paid tier and Hyper3D both have genuinely good stated privacy terms — but self-hosting sidesteps needing to trust and monitor any of it, and is cheaper at any volume once a GPU worker is already a required part of the architecture (which `map.md` has already decided). A commercial API remains a sensible **fallback/comparison point** — e.g. spin up a Meshy or Tripo3D trial to sanity-check TripoSR's/InstantMesh's output quality against a well-funded commercial bar during prototyping — but not the production default given the constraints as stated.
