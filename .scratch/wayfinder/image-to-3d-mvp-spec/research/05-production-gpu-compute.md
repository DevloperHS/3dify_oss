# Production GPU Compute for TripoSR: Serverless Inference vs. Traditional Autoscaled Instances

Scope: compare, against primary/official sources only, serverless/on-demand GPU inference platforms (Modal, RunPod Serverless, Replicate, Baseten) against traditional autoscaled cloud GPU instances (AWS EC2 G5/G4dn, GCP Compute Engine L4, Lambda Labs) as the production home for a TripoSR image-to-3D worker sitting behind a Redis/BullMQ job queue.

Research date: 2026-07-11

## Group 1: Serverless / On-Demand GPU Inference Platforms

### Modal

**Packaging model.** Modal has no model catalog — everything is arbitrary code. A function is wrapped in the `@app.function()` decorator with a `gpu=` argument and an `image=` argument; the image can be built programmatically (`modal.Image.debian_slim().pip_install(...)`), from an existing Dockerfile (`Image.from_dockerfile`), or pulled from a registry (Docker Hub, ECR, GCP Artifact Registry). This is a direct fit for TripoSR's custom PyTorch + CUDA + HuggingFace-checkpoint pipeline. ([modal.com/docs/guide](https://modal.com/docs/guide), [modal.com/docs/guide/gpu](https://modal.com/docs/guide/gpu))

**Pricing.** Modal's pricing page lists per-second rates: "Nvidia T4: $0.000164 / sec", "Nvidia L4: $0.000222 / sec", "Nvidia A10: $0.000306 / sec" (Modal calls it "A10", not "A10G"). The Starter plan includes "$30 / month free credits," Team includes "$100 / month free credits." ([modal.com/pricing](https://modal.com/pricing))

**Cold start.** Modal's own guide: "Containers boot in about one second," but initialization work (loading model weights) can extend total cold start "from minutes to seconds" only after optimization — implying unoptimized large-model cold starts land in the minutes range. Modal's dedicated GPU-snapshot feature addresses this directly: with GPU memory snapshots "the same Function can now take as low as 2s," and a real example (Ministral 3B) went from "~118s to ~12s" median cold start. ([modal.com/docs/guide/cold-start](https://modal.com/docs/guide/cold-start), [modal.com/blog/gpu-mem-snapshots](https://modal.com/blog/gpu-mem-snapshots), [modal.com/blog/mistral-3](https://modal.com/blog/mistral-3))

**Scale-to-zero.** "By default, Modal Functions will scale to zero when there are no inputs to process… you pay nothing for compute while idle." `min_containers` is an opt-in override to keep containers warm (and billed) continuously; the true-zero-cost behavior is the out-of-the-box default (`min_containers=0`). ([modal.com/docs/guide/scale](https://modal.com/docs/guide/scale))

### RunPod Serverless

**Packaging model.** Distinct from RunPod's "Pods" (rented VMs). Serverless requires a custom Docker image plus a `handler(event)` Python function as the entry point; RunPod's own base images bundle CUDA/Python/uv. This is a standard "write your own container" workflow, well suited to TripoSR. ([docs.runpod.io/serverless/overview](https://docs.runpod.io/serverless/overview), [github.com/runpod-workers/worker-template](https://github.com/runpod-workers/worker-template))

**Pricing.** RunPod's pricing page shows hourly-equivalent Serverless rates by GPU tier: 24GB tier "L4/A5000/RTX3090/MIG 24GB" at $0.69/hr, 16GB tier "A4000/A4500/RTX4000/RTX2000" at $0.58/hr, and 24GB "RTX 4090" at $1.10/hr. ([runpod.io/pricing](https://www.runpod.io/pricing)) RunPod's docs clarify the actual billing granularity: "pay-per-second pricing with no upfront costs," billing "from when a worker starts until it fully stops, rounded up to the nearest second." ([docs.runpod.io/serverless/pricing](https://docs.runpod.io/serverless/pricing)) No platform doc publishes a distinct $/sec figure separate from dividing the hourly rate by 3600.

**Cold start.** RunPod's marketing states "RunPod Serverless runs AI inference with sub-200ms FlashBoot cold starts" ([runpod.io/product/serverless](https://www.runpod.io/product/serverless)), and the FlashBoot launch post claims cold starts as low as ~1 second. These figures are best read as best-case, cache-optimized numbers for models RunPod has already staged — the docs are more conservative for the general case: "Cold start time measures how long it takes to wake up a worker, which includes the time needed to start the container, load the model into GPU VRAM, and get the worker ready to process a job," and "If cold start exceeds 7 minutes, the worker is marked unhealthy" (extendable via `RUNPOD_INIT_TIMEOUT`). RunPod's own guidance for cutting cold starts on large models is to use cached models or network volumes so weights don't have to be re-downloaded every cold start. ([docs.runpod.io/serverless/troubleshooting](https://docs.runpod.io/serverless/troubleshooting), [docs.runpod.io/serverless/storage/network-volumes](https://docs.runpod.io/serverless/storage/network-volumes), [runpod.io/blog/introducing-flashboot](https://www.runpod.io/blog/introducing-flashboot-serverless-cold-start))

**Scale-to-zero.** Endpoint config docs distinguish "Active workers" — "Minimum number of workers that remain warm and ready at all times… incur charges continuously, including when idle" (this is the 24/7-cost, zero-cold-start option) — from the default "Flex" behavior, which scales to zero after an idle timeout (default 5 seconds) with no manual configuration required beyond simply not setting active workers. ([docs.runpod.io/serverless/endpoints/endpoint-configurations](https://docs.runpod.io/serverless/endpoints/endpoint-configurations))

**Note on TripoSR specifically.** RunPod has a guide/blog article on running TripoSR on RunPod, but it walks through building a custom Docker container yourself ("Deploy a container optimized for 3D gen, loading TripoSR and process images to meshes") — it is not a one-click RunPod Hub template. ([runpod.io/articles/guides/generating-3d-models-with-tripos-gpu-platform](https://www.runpod.io/articles/guides/generating-3d-models-with-tripos-gpu-platform))

### Replicate

**Packaging model.** Replicate's deployment path is Cog, an open-source tool that "makes it easy to put a machine learning model in a Docker container." A `cog.yaml` defines the build (Python version, packages, system deps) and a `predict.py` defines a `Predictor` class; `cog push` builds and publishes the resulting container to Replicate. This supports arbitrary PyTorch/custom inference code, not just a model catalog. ([replicate.com/docs/guides/build/push-a-model](https://replicate.com/docs/guides/build/push-a-model), [github.com/replicate/cog](https://github.com/replicate/cog))

**Pricing.** Replicate's pricing page lists a per-second T4 rate of "$0.000225/sec" (~$0.81/hr). Replicate does not publish an L4 or A10G tier on its standard hardware price list — the next tier above T4 is L40S (48GB), well beyond TripoSR's ~6-8GB need, making T4 the natural fit. ([replicate.com/pricing](https://replicate.com/pricing))

**Cold start.** Replicate's own explanation: "requiring the fetching and loading of several gigabytes of code for some models... in some cases this process can take several minutes." A dedicated blog post shows Replicate has specifically optimized cold boots for *fine-tunes of officially supported base models* down to "less than one second," but this is called out as a special case, not the general behavior for an arbitrary custom Cog model like TripoSR. ([replicate.com/docs/how-does-replicate-work](https://replicate.com/docs/how-does-replicate-work), [replicate.com/blog/fine-tune-cold-boots](https://replicate.com/blog/fine-tune-cold-boots))

**Scale-to-zero / billing nuance — important caveat.** Replicate's billing docs draw a sharp line: for **public/official models**, "you only pay for the time it's active processing your requests. Setup and idle time for the model is free." For **private models and Deployments** (the path a custom TripoSR container would use), "we charge for all the time deployment instances are online: the time they spend setting up; the time they spend idle, waiting for requests; and the time they spend active, processing your requests" — i.e., idle time between requests is **billed**, not free, unless the "fast-booting fine-tune" exception applies. ([replicate.com/docs/topics/billing](https://replicate.com/docs/topics/billing)) This materially changes Replicate's cost profile for a custom low-traffic model compared to Modal/RunPod/Baseten's default free-idle behavior — a Replicate Deployment must be explicitly scaled down (min instances) to avoid paying for 24/7 idle capacity.

**Note on TripoSR specifically.** A community-published model, `camenduru/tripo-sr`, previously existed on Replicate (indexed by third-party model catalogs), but the page returned HTTP 404 when checked directly during this research session — it appears to be removed or no longer maintained, and was never an official/Replicate-maintained listing. There is no official TripoSR model on Replicate's catalog; a production deployment would mean pushing your own Cog container.

### Baseten

**Packaging model.** Baseten's packaging standard is Truss, described as "our open source standard for packaging and serving models built in any framework," used for both open-source and fully custom models. ([baseten.co/pricing](https://www.baseten.co/pricing/))

**Pricing.** Baseten's pricing page gives per-minute Dedicated Deployment rates: T4 (16 GiB) "$0.01052/min" (~$0.6312/hr), L4 (24 GiB) "$0.01414/min" (~$0.8484/hr), A10G (24 GiB) "$0.02012/min" (~$1.2072/hr). ([baseten.co/pricing](https://www.baseten.co/pricing/))

**Cold start.** Baseten's performance docs are candid about the scale of the problem and their mitigation: "The slowest part of a cold start is loading model weights, which can reach hundreds of gigabytes... Baseten is able to cold start most models in seconds and the largest models in a handful of minutes." A concrete example given: Stable Diffusion XL "can take a few minutes to boot up without caching, but with caching, it takes just under 10 seconds," via the Baseten Delivery Network (BDN), a caching layer that mirrors model weights to Baseten's own blob storage on first deploy. ([docs.baseten.co/performance/cold-starts](https://docs.baseten.co/performance/cold-starts), [baseten.co/blog/baseten-delivery-network-fast-cold-starts-big-models](https://www.baseten.co/blog/baseten-delivery-network-fast-cold-starts-big-models/)) TripoSR's checkpoint is far smaller than the 70B-parameter examples Baseten's docs are written around, so these cited multi-minute cold starts represent a worst case, not a typical case, for this workload.

**Scale-to-zero.** Baseten's pricing page states plainly: "you do not pay for idle time – you only pay for the time your model is using compute," and that users retain "complete autoscaling control," implying scale-to-zero is available and configurable rather than always-on by default. ([baseten.co/pricing](https://www.baseten.co/pricing/))

**Note on TripoSR specifically.** No evidence was found of an official TripoSR listing in Baseten's model library; deployment would go through Truss as a custom model, same as Modal/RunPod/Replicate.

### Excluded: Fal.ai

Fal.ai's public pricing page ([fal.ai/pricing](https://fal.ai/pricing)) lists GPU rental rates only for high-end tiers (RTX PRO 6000 96GB and up — B300/B200/H200/H100) with no published 16-24GB entry-level tier, and its custom-deployment offering ("Deploy your own app on our competitively priced GPU fleet") has no documented pricing, container model, or cold-start figures — the page directs custom-deployment inquiries to `support@fal.ai`. Per this task's instruction to skip platforms without a real, findable official pricing page for the relevant tier, Fal.ai is excluded from the detailed comparison above.

## Group 2: Traditional Cloud GPU Instances with Autoscaling

### AWS EC2 G5 and G4dn

**Instance fit.** `g5.xlarge` provides "1x NVIDIA A10G Tensor Core GPU" with "24 GB of memory per GPU," 4 vCPUs, 16 GiB RAM. `g4dn.xlarge` provides 1x NVIDIA T4 (16GB), 4 vCPUs, 16 GiB RAM. ([aws.amazon.com/ec2/instance-types/g5](https://aws.amazon.com/ec2/instance-types/g5/), [aws.amazon.com/ec2/instance-types/g4](https://aws.amazon.com/ec2/instance-types/g4/))

**Pricing.** AWS's own EC2 On-Demand pricing pages render their tables client-side via JavaScript and could not be scraped directly through automated fetch during this research session (repeated fetches returned only page chrome, no table data). The figures below are cross-verified across two independent AWS Price-List-API-sourced trackers (Vantage and economize.cloud) which agreed exactly on the value, and are consistent with the fact that AWS markets G5 as offering "up to 40% better price performance for machine learning inference compared to G4dn instances" ([aws.amazon.com/ec2/instance-types/g5](https://aws.amazon.com/ec2/instance-types/g5/)) — a claim only consistent with G5 being priced roughly 2x G4dn while delivering more than 2x the inference throughput:
- `g5.xlarge`: **$1.006/hr** on-demand, US East (N. Virginia)
- `g4dn.xlarge`: **$0.526/hr** on-demand, US East (N. Virginia)

Treat these two numbers as needing final confirmation directly against [aws.amazon.com/ec2/pricing/on-demand](https://aws.amazon.com/ec2/pricing/on-demand/) in a browser, since AWS's own page could not be fetched as static content.

**Autoscaling.** AWS's managed product here is EC2 Auto Scaling Groups (ASG) — a fully managed service where the developer defines a launch template and scaling policies (target tracking, step scaling, or scheduled scaling); AWS's docs confirm ASGs can be created with `--min-size 0 --max-size 0 --desired-capacity 0`, i.e., scale-from-zero is supported. ([docs.aws.amazon.com/autoscaling/ec2/userguide/auto-scaling-groups.html](https://docs.aws.amazon.com/autoscaling/ec2/userguide/auto-scaling-groups.html)) However, this scale-up/scale-down logic (what metric triggers scale-out, how quickly to scale in, how to drain in-flight jobs) is something the developer must design and configure — AWS provides the scaling primitive, not a queue-aware autoscaler out of the box. **GPU-specific caveat:** AWS's EC2 service quotas system caps On-Demand vCPU usage per instance family group per Region ([docs.aws.amazon.com/AWSEC2/latest/UserGuide/ec2-resource-limits.html](https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/ec2-resource-limits.html)); community reports (AWS re:Post) consistently describe the "Running On-Demand G and VT instances" quota defaulting to 0 vCPUs for new accounts, meaning a quota increase request — with an unpredictable approval turnaround — is typically required before a G5/G4dn instance can be launched at all. This specific "defaults to 0" figure could not be pinned to an official AWS documentation page during this research session (only AWS's own community forum, not docs.aws.amazon.com) and should be verified against the account's actual Service Quotas console before relying on it.

**24/7 vs. traffic-window cost.**
- `g5.xlarge` 24/7: $1.006 × 730 hrs ≈ **$734/month**
- `g4dn.xlarge` 24/7: $0.526 × 730 hrs ≈ **$384/month**
- Low-traffic reality: EC2 On-Demand has no idle discount — every hour running costs the full rate whether processing a job or not. For an early-stage product, running 24/7 only makes sense once traffic is roughly continuous; otherwise the developer must build their own start/stop automation (e.g., a scheduled Lambda, or an ASG scaling policy tied to BullMQ queue depth) to approximate serverless-style cost savings, and even a well-tuned scale-down still leaves several minutes of idle-billed time per scale-in event (there is no per-second-idle-free behavior as on Modal/RunPod/Baseten).

([docs.cloud.google.com/compute/docs/gpus/gpu-machine-types](https://docs.cloud.google.com/compute/docs/gpus/gpu-machine-types))

## Cold-Start Implications for an Async Job-Queue Pipeline

The BullMQ architecture in this project is inherently asynchronous: a user uploads an image, a job is enqueued, and the user's client polls (or receives a websocket/push update) for job completion — nobody is blocked on an open HTTP connection waiting for the GPU call to return. This changes the cost/benefit calculus of cold starts substantially compared to a synchronous request/response API:

- **10-30 seconds of cold start is comfortably tolerable.** If a BullMQ worker calls a serverless GPU endpoint and that specific invocation happens to hit a cold container, the total round trip becomes ~20s (TripoSR's own compute, per the HuggingFace Space prototype numbers) plus another 10-30s of cold start — the user sees this as "still generating," the same UI state they'd see either way, with no protocol-level timeout at risk (BullMQ jobs don't have an HTTP-request-style short timeout by default; job timeout is whatever the developer configures on the queue). This is exactly the tradeoff Modal, RunPod, Replicate, and Baseten are each designed around — all four platforms explicitly favor scale-to-zero economics for exactly this kind of bursty/low-traffic workload, at the cost of an occasional cold start that a synchronous caller would find unacceptable but an async worker does not.
- **The true worst case (30s-2min) is more concerning but still survivable, with two caveats.** Per the platforms' own docs above, worst-case cold starts for a several-GB uncached checkpoint range from RunPod's 7-minute unhealthy-worker ceiling, to Baseten's "handful of minutes" for uncached large models, to Replicate's "several minutes" for uncached custom models. TripoSR's checkpoint is modest by these platforms' own standards (the multi-minute examples they cite are 70B-parameter LLMs or full SDXL pipelines, not a ~topology-reconstruction transformer), so a multi-minute worst case is unlikely in steady state — but it is realistic for the *very first* cold start after a deploy, or after a long idle period beyond whatever the platform's weight-cache retention window is. The mitigations are: (1) use each platform's model-caching feature (RunPod network volumes / cached models, Baseten's BDN, Modal's GPU memory snapshots) so only the *first-ever* cold start pays the multi-minute penalty and all subsequent cold starts are fast; and (2) set a generous but bounded BullMQ job timeout (e.g., 3-5 minutes) with a clear "still working" UI state, rather than assuming every request is warm.
- **Contrast with a synchronous API:** if this were a blocking HTTP endpoint (e.g., a browser `fetch` awaiting a direct response), even a 10s cold start risks tripping common gateway/proxy timeouts (many default around 30-60s) and definitely degrades perceived responsiveness; a 30s-2min cold start would be a hard failure for a synchronous caller. The queue absorbs exactly this risk, which is the core architectural reason a BullMQ-fronted pipeline is well matched to serverless GPU cold-start economics — pay-per-second, scale-to-zero billing has essentially zero downside for this project's actual UX shape.

## Integration Complexity Against the Redis/BullMQ Node.js Worker

- **Serverless platforms — HTTP/REST call from inside the BullMQ processor, no SDK required (though SDKs exist).** In all four serverless platforms researched, the natural integration point is: the BullMQ worker function (a plain Node.js process pulling jobs off Redis) makes an outbound HTTP call to the platform's inference endpoint when it picks up a job, then either polls a status endpoint or awaits a synchronous response, and writes the GLB result once it returns. Modal exposes deployed functions as HTTP endpoints callable from any language (no Python SDK required from Node, though one exists for Python-side authoring); RunPod Serverless is invoked via its REST job API (`/run` or `/runsync` plus a `/status` poll); Replicate is invoked via its REST API (create prediction, poll or webhook); Baseten is invoked via a REST API per deployed model. None of these require the Node.js BullMQ worker to run any platform-specific runtime — it's the same "call an HTTP API, wait or poll" pattern regardless of platform, which keeps the worker code and mental model consistent across all four options and cheap to swap between if pricing/reliability warrants a change later.
- **Traditional instances — the BullMQ worker (or something else) must own health/routing.** For AWS G5/G4dn, GCP L4, and Lambda Labs, there is no managed request-routing layer bundled with "just an instance." The project has to choose one of: (a) put a lightweight HTTP server directly on the GPU instance (the instance itself pulls jobs from Redis/BullMQ or exposes a REST endpoint the main worker calls) and put a load balancer or the BullMQ worker's own retry/health-check logic in front of it to route around instances that are down or still booting; or (b) run a Kubernetes-based setup (GKE) where a Service/Ingress does the routing, at the cost of adopting Kubernetes as an additional operational dependency. Either way this is materially more integration work than the serverless "call the API" pattern — the project would be building the job-routing and health-check logic that Modal/RunPod/Replicate/Baseten provide for free as part of their platform.

## Recommendation

**Start with Modal.** The working assumption — that serverless/pay-per-second GPU compute is the better fit than dedicated autoscaled instances at this stage — holds up against the numbers gathered:

- **Cost at low/unproven traffic is materially better than any traditional instance.** Modal's L4 rate of $0.000222/sec means a ~20-second TripoSR job (per the prototype's real round-trip numbers) costs well under a cent of raw compute, and idle time between jobs costs exactly $0 by default (`min_containers=0`). The cheapest comparable traditional instance (`g4dn.xlarge` at $0.526/hr) costs money every hour regardless of whether a job is running — for an early-stage product processing, say, a few dozen jobs a day, the traditional-instance 24/7 cost (~$384-734/month before any traffic-window optimization work) is not defensible next to Modal's near-zero idle cost.
- **Cold starts are tolerable for this architecture, and best-in-class among the options researched.** Modal's documented "~1s" container boot plus GPU memory snapshot examples showing multi-minute cold starts reduced to single-digit seconds gives real headroom against the "10-30s is fine, 30s-2min is the risk zone" analysis above — and Modal is the only one of the four serverless platforms with a named, documented technique (GPU memory snapshots) specifically aimed at cutting the exact failure mode (loading a multi-GB checkpoint onto a fresh GPU) this project cares about.
- **Packaging is a clean fit for TripoSR's custom pipeline.** Modal's `@app.function(gpu=..., image=...)` model with full custom-Docker-image support (including `Image.from_dockerfile`) handles TripoSR's specific requirement (PyTorch + a specific CUDA version + a custom Python script that loads a HuggingFace checkpoint and runs reconstruction) with no workaround needed — this is true of RunPod and Baseten too, but Modal's Python-native decorator model is the lowest-friction of the three for a team not already invested in writing a `Dockerfile` + `handler.py` (RunPod) or a Truss config (Baseten) by hand.
- **Genuine caveats that keep this from being a slam dunk:** (1) Modal's per-second rate for L4/A10 is not the cheapest of the four serverless options — RunPod's equivalent tiers are nominally cheaper per hour ($0.69/hr vs. Modal's L4 ~$0.80/hr-equivalent, $0.58/hr vs. Modal's T4 ~$0.59/hr-equivalent) — so if cost becomes the dominant constraint at higher volume, RunPod Serverless is worth a second look, with the tradeoff that RunPod's cold-start story is less precisely documented for the large-uncached-model case (its 7-minute unhealthy-worker ceiling is a wider worst-case band than anything Modal publishes). (2) Replicate is the weakest fit specifically because of its idle-billing rule for private/custom deployments — a TripoSR container on Replicate would need explicit min-instance tuning to avoid being billed for idle time the same way a traditional instance would, undercutting the core "pay only for what you use" appeal that makes serverless attractive here in the first place. (3) None of the four platforms offer TripoSR as a one-click/pre-packaged model (a community Replicate listing existed but returned 404 during this research and was never official) — every option requires packaging TripoSR's own inference code into that platform's container format, so the "packaging effort" delta between the four serverless options is really about *how much boilerplate* their packaging model demands, not whether packaging is needed at all.

If, once real usage data comes in, sustained/predictable traffic makes 24/7 utilization likely, that's the point to revisit a traditional autoscaled instance (most plausibly GKE with L4 nodes, given it at least has a documented scale-from-zero path via KEDA) — but for an unproven, early-traffic product, the integration simplicity, near-zero idle cost, and documented cold-start mitigations make Modal the stronger starting point than building and operating a self-managed autoscaling GPU fleet.

## Data Points Not Verified Against a Directly-Fetchable Primary Source

- **AWS `g5.xlarge` ($1.006/hr) and `g4dn.xlarge` ($0.526/hr) on-demand rates** — AWS's own pricing pages (`aws.amazon.com/ec2/pricing/on-demand/` and the G5/G4dn instance-type pages) are JavaScript-rendered and did not return tabular pricing data to automated fetch; figures were cross-verified across two independent AWS-Price-List-API-sourced trackers instead of read directly off `aws.amazon.com`.
- **GCP `g2-standard-4` (~$0.7068/hr) on-demand rate** — same issue: `cloud.google.com/compute/gpus-pricing` and related GCP pricing pages returned truncated content to automated fetch; figure is a search-snippet/aggregator cross-reference, not a direct read of Google's own table.
- **AWS "Running On-Demand G and VT instances" quota defaulting to 0 vCPUs for new accounts** — well-corroborated across AWS re:Post community threads, but not pinned to an official `docs.aws.amazon.com` page stating the specific default value during this research session; the general quota mechanism (and where to check/request an increase) is confirmed via AWS's official EC2 service-quotas doc.
- **RunPod's exact per-second $ rate** (as opposed to the published hourly-equivalent rate) — RunPod's docs confirm billing is per-second, but no page publishes a $/sec figure distinct from hourly-rate ÷ 3600.
